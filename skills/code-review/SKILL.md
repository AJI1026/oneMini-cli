# Code Review

## 流程

1. 用 `git diff` 或 read 定位变更范围
2. 按下列维度审查，**先列问题再列优点**
3. 每条问题标注：**严重程度**（blocker / major / minor / nit）与 **具体文件:行**

## 审查清单

| 维度 | 检查点 |
|------|--------|
| 正确性 | 边界条件、错误处理、并发/竞态、资源泄漏 |
| 安全 | 注入、路径穿越、密钥泄露、权限绕过 |
| 设计 | 职责单一、命名、重复代码、API 稳定性 |
| 测试 | 关键路径是否有测试；回归风险 |
| 可维护性 | 复杂度、注释是否必要、与项目风格一致 |

## 输出格式

```markdown
## 总结
（1-2 句总体评价）

## 必须修复
- [blocker] ...

## 建议改进
- [major/minor] ...

## 做得好的地方
- ...
```

## 原则

- 不吹毛求疵无关风格（除非项目有明确规范）
- 可运行的改进建议附带示例代码或具体改法
- 无法确认的行为标注「需确认」而非臆断
