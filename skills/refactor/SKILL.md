# Refactor

## 前置

1. 确认有测试或能运行的验证命令（`cargo test` / `npm test` 等）
2. 无测试时：先列出**行为不变**的验收标准
3. 建议先创建 git 检查点（若用户允许）

## 原则

- **小步提交**：每步可独立验证
- **行为等价**：不改对外 API 语义（除非用户明确要求）
- **先测后改**：每步重构后跑验证

## 常见手法（按需选用）

| 手法 | 适用 |
|------|------|
| 提取函数/模块 | 重复逻辑、过长函数 |
| 重命名 | 误导性命名 |
| 搬移 | 错误层级的依赖 |
| 简化条件 | 嵌套过深 |

## 流程

1. 说明重构目标与影响范围
2. 列出 3-5 步计划
3. 逐步执行，每步后验证
4. 总结：改了什么、如何验证、剩余技术债（如有）

## 禁止

- 混合功能变更与重构（应拆分）
- 大规模重写无测试覆盖
- 修改与目标无关的文件
